# System Maintenance Lifecycle Workflow

这是 Rhea 自有系统维护生命周期 Workflow，用于把模糊的系统、应用、Runtime 或依赖问题推进到可审批、可回滚、可验证的结果。

## When To Use

- 用户提出软件安装、卸载、升级、配置或故障排查。
- Runtime、Agent、包管理器或开发环境不可用。
- 任务可能涉及权限、系统设置、PATH、服务、启动项、删除或外部写入。

## Workflow

### 1. Intake

识别任务类型：安装、卸载、依赖、PATH、权限、Runtime、进程、磁盘、包导入或失败恢复。

### 2. Scope

明确系统平台、目标软件、项目目录、允许动作、禁止动作、敏感信息边界和输出用途。

### 3. Read-Only Check

优先生成只读检测任务，例如版本、路径、安装状态、配置摘要、日志摘要。只读检测也要限定范围。

### 4. Risk Review

检查管理员权限、sudo、删除、卸载、PATH、服务、启动项、未知脚本、外发数据和回滚条件。

### 5. Approval

需要写入或高风险动作时，生成用户审批请求。Rhea 不替用户批准。

### 6. Execute By Runtime

真实执行交给目标 Runtime 或外部 Runtime。任务包必须包含目标、范围、禁止事项、审批状态、证据要求和失败回收。

### 7. Evidence Review

对照原始验收标准，检查版本、路径、退出状态、日志、截图、失败项、未完成项和是否触碰禁止事项。

### 8. Report

输出维护报告：结论、做了什么、证据、风险、未完成项、下一步和是否建议记忆。

### 9. Memory Candidate

只有非敏感、可复用、来源清楚、边界明确的经验才建议写成记忆候选。

## Completion Criteria

- 问题类型清楚。
- 只读检测优先。
- 高风险动作有审批点。
- 执行层任务可复制。
- evidence 能覆盖验收标准。
- 维护报告不夸大完成程度。
