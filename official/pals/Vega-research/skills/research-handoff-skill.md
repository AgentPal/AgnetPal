# research-handoff

## Purpose

Package research findings for Mira, Atlas, PalSmith, Quinn, Rhea, Nova, Morgan, or Harper without losing evidence, risk, or decision context.

## When to use

Use when Vega's research supports another Pal's implementation, governance, quality review, system-risk review, product decision, document work, or writing task.

## Inputs

Research brief, source inventory, evidence matrix, confidence notes, open questions, target Pal, and expected next action.

## Required context

Default-Pal collaboration boundaries and the current central Pal roster.

## Process

1. Confirm target Pal and reason using AI judgement.
2. Summarize user goal and research question.
3. Attach source IDs and evidence matrix summary.
4. Mark confidence, uncertainty, risks, and user decisions.
5. State what the target Pal should do and should not do.
6. Include required evidence to return.

## Output

Context Packet or Runtime Task Package fragment for the selected Pal or execution layer.

## Quality bar

The handoff must be usable without rereading every source, while preserving links back to the evidence.

## User confirmation points

Ask before sharing sensitive context, private materials, or low-confidence conclusions with another Pal or Runtime.

## No-code boundary

Handoff is a coordination artifact. It does not execute the target Pal's work or create runtime orchestration.

## Common mistakes

- Sending conclusions without source IDs.
- Hiding low confidence.
- Treating a handoff as automatic routing.
- Sharing too much sensitive context.

## Example

"Atlas receives a technical option research packet with criteria, source IDs, tradeoffs, uncertainty, file boundaries, and implementation questions."
