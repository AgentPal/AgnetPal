# source-discovery

## Purpose

Guide Runtime toward source sets that are useful, diverse, and appropriate for the user's research decision.

## When to use

Use after question framing and before credibility scoring when a source set does not yet exist.

## Inputs

Search plan, required source types, known sources, freshness need, user constraints, and forbidden sources.

## Required context

Search strategy, source credibility criteria, and source inventory template.

## Process

1. Identify the minimum source categories for the task.
2. Prefer primary and official sources for factual claims.
3. Add reputable secondary sources for interpretation and comparison.
4. Add counter-sources or limitation sources.
5. Request metadata: title, URL, publisher, date, type, author if available, and access date.
6. Reject sources that cannot support the claim type.

## Output

Source discovery request for Runtime plus expected inventory rows.

## Quality bar

The discovered set must cover the core question, not just produce many links.

## User confirmation points

Ask before using private accounts, workspace connectors, paid reports, or user-uploaded confidential sources.

## No-code boundary

Vega does not fetch sources by itself. Runtime or external tools provide evidence under current permissions.

## Common mistakes

- Accepting listicles as equivalent to primary evidence.
- Ignoring publication date.
- Missing source type diversity.
- Using community sentiment as proof of technical fit.

## Example

For a library choice, discover official docs, repo activity, release notes, license, security advisories, adoption evidence, and migration reports.
