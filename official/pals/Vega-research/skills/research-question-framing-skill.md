# research-question-framing

## Purpose

Turn broad or ambiguous research requests into answerable primary questions, subquestions, evidence needs, and exclusion rules.

## When to use

Use before any medium or deep research, technical comparison, market scan, policy check, or source review.

## Inputs

User goal, decision to support, context, target audience, known constraints, time horizon, location, version, and output format.

## Required context

Research role knowledge, source credibility knowledge, and relevant user/project constraints.

## Process

1. Identify the decision or understanding the research must support.
2. Write one primary research question.
3. Add three to seven subquestions.
4. Define inclusion and exclusion rules.
5. Define freshness and geography/version boundaries.
6. Define what would count as enough evidence.

## Output

Question frame with primary question, subquestions, source types, boundaries, and expected evidence.

## Quality bar

Each question must be answerable by evidence. The frame must separate descriptive, comparative, causal, and recommendation questions.

## User confirmation points

Ask for confirmation when scope, geography, time window, product tier, or decision criteria would change the outcome.

## No-code boundary

This skill creates a research frame only. It does not run searches, scrape pages, or build tools.

## Common mistakes

- Asking a question that already assumes the answer.
- Mixing product preference with factual evidence.
- Ignoring time and version boundaries.
- Framing a regulated-domain decision as casual advice.

## Example

Poor frame: "Which is better?" Better frame: "For a Windows desktop agent workspace in 2026, which option better supports local file editing, source traceability, and low setup risk?"
