# research-intake

## Purpose

Classify a user research request, define depth, risk, freshness need, source need, and final deliverable before any Runtime work begins.

## When to use

Use when a user asks Vega to research, compare, verify, summarize sources, find current information, or prepare evidence for another Pal.

## Inputs

User question, intended use, audience, deadline, freshness need, known sources, forbidden sources, sensitivity level, and desired output shape.

## Required context

Vega role boundary, source verification protocol, current contact registry if collaboration may be needed, and any user-provided materials.

## Process

1. Restate the research goal in one sentence.
2. Identify research type: quick check, deep research, source review, competitive research, technical option research, knowledge distillation, or handoff.
3. Mark freshness need: current, recent, stable background, or user-provided-only.
4. Identify risk: low, medium, high, or regulated domain.
5. Name missing user decisions.
6. Produce a Research Task Package for Runtime when external evidence is needed.

## Output

Research intake note with goal, scope, depth, source need, evidence format, user confirmations, Runtime evidence needed, and no-code boundary.

## Quality bar

The intake must prevent vague browsing, hidden assumptions, and unsupported conclusions. It must say when web research is not run.

## User confirmation points

Ask before using sensitive material, paid/private sources, high-risk external actions, or broad research beyond the user's requested scope.

## No-code boundary

Vega plans and reviews research. Runtime may browse or read sources, but Vega does not add crawlers, scripts, scanners, validators, installers, or UI.

## Common mistakes

- Treating every request as deep research.
- Accepting "latest" without live evidence.
- Treating source count as quality.
- Omitting user decision needs.

## Example

Input: "Compare two agent frameworks for our app." Output: "This is technical option research. Need current docs, release status, integration constraints, license, risks, and an evidence matrix before recommendation."
