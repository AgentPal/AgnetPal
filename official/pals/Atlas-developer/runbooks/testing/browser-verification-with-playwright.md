# Browser Verification With Playwright Runbook

这个 Runbook 用于把 UI 验收交给具备 Playwright、浏览器自动化或前端测试能力的 Runtime / 外部 Runtime。

Atlas 不直接运行浏览器命令，不读取登录态，不操作真实生产数据。Atlas 负责把 UI 验收标准写清楚，并要求执行层返回可复验 evidence。

## When To Use

- 用户故事需要浏览器路径验证。
- UI 修复需要截图或交互证据。
- 前端回归需要确认关键流程。
- 页面能打开但功能是否可用仍不确定。

## Browser Task Context

交接给执行层前写清：

- 目标页面或本地 URL。
- 测试环境和账号状态。
- 用户路径。
- 必须验证的交互。
- 禁止触碰的真实数据或生产动作。
- 截图、日志、trace 或视频要求。
- 失败时需要记录的信息。

## Handoff Structure

```text
## 浏览器验证目标

## 环境

## 用户路径

## 验收标准

## 禁止事项

## 建议工具

## Flaky 处理

## Evidence 要求

## 完成汇报格式
```

## Verification Principles

- 页面能打开不等于功能可用。
- 按钮存在不等于流程成功。
- 截图应覆盖关键状态，而不是只截首页。
- 失败日志要包含步骤、期望、实际结果和可疑原因。
- flaky 测试要复跑或标明不稳定条件。

## Flaky Handling

如果测试不稳定，执行层应报告：

- 失败频率。
- 失败步骤。
- 是否与等待、网络、动画、数据状态或定位器有关。
- 是否需要更可靠的选择器或测试数据隔离。

Atlas 不接受“偶尔失败但应该没事”作为完成证据。

## Evidence Review

Atlas 检查：

- 是否覆盖完整用户路径。
- 是否有截图、trace、日志或明确手工步骤。
- 是否说明测试环境。
- 是否区分功能成功、视觉正常和数据正确。
- 是否触碰了禁止事项。

涉及真实生产账号、支付、删除、发送消息、发布或外部写操作时，必须先要求用户确认。
