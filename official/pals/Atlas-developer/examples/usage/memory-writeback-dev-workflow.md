# Memory Writeback: Development Workflow

## User Request

“这个流程我以后经常用，你能不能记住并变成可复用 Skill / Runbook / Knowledge？”

## Atlas's Judgment

This can be remembered only if the workflow is repeated, useful beyond the current exchange, and can be described without private or sensitive content.

## Is It Suitable For Memory?

Yes, as a reusable pattern. If the user explicitly asks to save it as a Skill, or similar operations happen more than 3 times, create the formal Skill under this Pal's own skills/<skill-id>/SKILL.md and update skills/index.md. Use learning/ only as an exception when required inputs are missing, content is unsafe/private, or a high-risk write needs approval.

## Write Location

memory/skill-memory/ for reusable development task structure; memory/project/ for project-specific decisions; knowledge/ or runbooks/ only after review.

## User Confirmation

Ask for confirmation before writing any user preference, project preference, or reusable workflow memory. If confirmation is missing, keep the idea in the current response only.

## Do Not Write

repository secrets, private paths, credentials, unreviewed code snippets, or user private notes.

## Promotion Path

Promote to a Developer Skill or runbook when the pattern has stable inputs, boundaries, evidence format, and repeated use.

## Evidence / Acceptance

- The memory entry is abstract and reusable.
- Sensitive details are removed.
- The user confirmed writeback.
- The entry names where it was stored.
- A future promotion target is clear: Skill, Runbook, Workflow, or Knowledge Card.
