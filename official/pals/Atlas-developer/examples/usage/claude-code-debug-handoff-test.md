# Claude Code Debug Handoff Test

## 用户请求

Codex 修一个构建错误失败了，我想让 Claude Code 帮我单独分析错误原因。

## 协作对象判断

适合交给 Claude Code。

原因：这是失败诊断任务，需要读取上下文、分析日志、形成假设、定位根因，并可能给出最小修复建议。Claude Code 适合长上下文代码理解和多轮调试。

Atlas 不直接运行命令，不直接修改文件。Atlas 只生成 Debug Handoff。

## 交接上下文包

目标：
分析 Codex 未能修复的构建错误，找出根因，并给出最小修复方案。

项目路径：
由用户提供。

当前状态：
Codex 已尝试修复但失败，需要 Claude Code 独立分析。

## 必读文件

- `AGENTS.md`
- 项目 README。
- 构建配置文件。
- 失败相关源码文件。
- Codex 完成报告或失败报告。

## 错误日志字段

请用户或 Codex 提供：

- 构建命令。
- 完整错误摘要。
- 首个失败堆栈。
- 最近修改文件。
- 失败前后是否安装依赖或改配置。

## 禁止事项

- 不安装新依赖，除非先说明原因并等待确认。
- 不删除测试绕过失败。
- 不修改 `.env`、凭据、真实数据或发布配置。
- 不做大范围重构。
- 不 git push。

## Claude Code 预期输出

- 是否能复现。
- 1 到 3 个根因假设。
- 最可能根因。
- 最小修复建议。
- 需要修改的文件。
- 建议测试命令。
- 风险和回滚说明。

## 验收标准

- 根因解释能对应错误日志。
- 修复范围最小。
- 没有越过禁止事项。
- 给出回归测试或未测试原因。

## 模型 / 强度建议

- 建议模型：Claude Code 当前可用强模型。
- 推理强度：high。
- 任务长度：中。

## 无法复现时怎么报告

Claude Code 应停止并报告：

- 缺少哪些日志或文件。
- 已检查哪些线索。
- 为什么无法复现。
- 下一步需要用户或 Codex 提供什么。

不允许编造复现结果。

## Atlas 如何复验

Atlas 收到结果后检查：

- 是否解释根因。
- 是否引用日志或文件线索。
- 是否给出最小修复路径。
- 是否要求测试 evidence。
- 是否触碰安装、删除、凭据、发布或 git push 风险。
