# Composite Implementation Stage Package

## User Request

"Vega found the sources and Harper drafted the copy. Create the implementation-stage package for the static page."

## When this Pal is a good candidate

Atlas is a good candidate because the requested stage is implementation-shaped: file scope, build/check commands, rendering evidence, and handoff to verification.

## When this Pal should not take the whole task

Atlas should not redo the research or rewrite the copy unless the accepted prior stage output is incomplete and the conductor asks for repair.

## Task Judgement

- domain_focus: implementation-stage planning.
- deliverables: file-scope package, implementation steps, checks, evidence contract.
- work_stages: accept prior content, map files, define edits, define verification.
- capability_needs: development planning, file boundaries, visual/check evidence.
- pal_candidates: Atlas for implementation candidate, Quinn for verification candidate, Mira for conductor summary candidate.
- runtime_candidates: file-editing Runtime, local preview if available, screenshot or diff evidence if available.
- verification_needs: rendered artifact, diff scope, acceptance against accepted content.

## Context Needs

Read the accepted research summary, accepted copy, target file structure, design constraints, and allowed write paths. Do not read unrelated source material.

## Suggested Task Package

- Goal: turn accepted content into an implementation-ready package.
- Allowed writes: target page files only.
- Steps: map content sections, edit files, run available checks, capture evidence, list not-run checks.
- Output: changed files, commands, screenshots if available, residual risks.

## Good Response Example

Atlas keeps the page stage bounded, names exact files, preserves accepted content, and requires runtime evidence before completion.

## Bad Response Example

Atlas treats the earlier research as optional, changes the product claims, or says the page is done without preview or diff evidence.

## Verification / Acceptance Criteria

- Prior-stage content is preserved or deviations are justified.
- File scope is explicit.
- Checks and not-run items are reported.
- Verification is separated from implementation.
