# multi-thread-agent-dispatch

运行时 Skill 入口请看 SKILL.md；README.md 仅作为人类阅读说明。


中文名：多线程 / 多 Runtime 开发派发

## 用途

把中大型开发任务拆成 2-4 个互不重叠的执行线程，供 Codex、Claude Code 或其他执行层并行处理。

## 适用场景

- 任务涉及文档、模板、测试、代码审查等多个独立面。
- 一个执行窗口上下文过大。
- 用户希望多线程并行推进。
- 不同任务适合不同模型或推理强度。

## 不适合场景

- 单文件小修。
- 需求还没定范围。
- 必须严格串行执行的迁移。
- 多线程会争用同一核心文件。

## 输入信息

- 总目标。
- 仓库路径。
- 全局必读资料。
- 可修改范围和禁止范围。
- 已知依赖顺序。
- 期望线程数量。
- 可用 Runtime、模型和工具能力。

## 处理流程

1. 判断是否适合并行。
2. 按可闭环能力拆线程，不按字段、按钮或碎片需求机械拆分。
3. 每个线程设置独立目标、文件范围、禁止事项、验收标准和完成报告格式。
4. 避免两个线程修改同一文件区域。
5. 给每个线程建议模型、推理强度和任务长度。
6. 设置最终整合与复验线程。

## 输出格式

```text
## 线程 N：标题
## 目标
## 必读资料
## 修改范围
## 禁止事项
## 验收标准
## 建议模型
## 推理强度
## 任务长度
## 完成汇报格式
```

## 验收标准

- 线程之间文件范围不冲突。
- 每个线程都能独立执行和汇报。
- 有最终复验或整合安排。
- 进度按发布贡献判断，不按代码行数或文档数量判断。

## 风险边界

不要把多个线程派到同一文件同一区域。不要让弱模型承担开放式架构判断。不要把高风险命令藏在线程任务里。

## 与 Atlas 其他 Skill / Runbook 的关系

- 使用 `context-engineering` 为每个线程准备上下文。
- 使用 `test-plan-writer` 为每个线程设置验收。
- 使用 `execution-evidence-review` 汇总复验。
