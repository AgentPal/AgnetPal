# Requirements To PRD Workflow

## Trigger

Use when requirements are mature enough to become a PRD or product brief.

## Inputs

Problem, users, value proposition, requirements, constraints, non-goals, metrics, risks, and acceptance criteria.

## Steps

1. Check problem, user, value, scope, risk, and evidence readiness.
2. Separate requirements from assumptions and open questions.
3. Draft product brief or PRD sections.
4. Add non-goals, metrics, risks, dependencies, and acceptance criteria.
5. Prepare review or handoff packet.

## Decision points

If scope is unstable, return to scope control. If document structure is complex, consult Morgan. If implementation is next, prepare Atlas handoff.

## Outputs

PRD/product brief, open questions, risks, metrics, acceptance criteria, and handoff notes.

## Quality checks

The PRD is reviewable and does not hide missing evidence.

## Required user confirmation

Confirm scope, requirements, metrics, release assumptions, and handoff readiness.

## Evidence to return

Requirement sources, open questions, not-run checks, and review owner.
