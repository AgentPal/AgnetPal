# Product Proposal To Dev Package

## User Request

"Write the product proposal, then prepare what the developer needs to build it."

## When this Pal is a good candidate

Nova is a good candidate for the product proposal stage: problem, users, scope, success criteria, and handoff requirements.

## When this Pal should not take the whole task

Nova should not own the implementation package if technical file scope and checks are required. That stage needs a development owner candidate.

## Task Judgement

- domain_focus: product proposal and handoff.
- deliverables: product brief and development handoff inputs.
- work_stages: product framing, scope definition, acceptance criteria, implementation-stage package candidate.
- capability_needs: product strategy, requirements, handoff clarity.
- pal_candidates: Nova for product proposal; Atlas candidate for implementation package; Quinn candidate for acceptance review.
- runtime_candidates: file-editing Runtime only after an implementation package is approved.
- verification_needs: product assumptions, acceptance criteria, later implementation evidence.

## Context Needs

Need user problem, constraints, desired outcome, target platform, non-goals, and evidence already available.

## Suggested Task Package

- Goal: produce a product brief and handoff, not implementation.
- Output: problem, users, requirements, non-goals, acceptance criteria, open engineering questions.
- Next stage: development package by a selected owner candidate if approved.

## Good Response Example

Nova writes a product brief and clearly hands off implementation questions without pretending the build has started.

## Bad Response Example

Nova writes a product brief and then tells the Runtime to build files with no technical owner judgement.

## Verification / Acceptance Criteria

- Product scope and non-goals are explicit.
- Implementation stage is separated.
- Acceptance criteria are testable.

## Notes
