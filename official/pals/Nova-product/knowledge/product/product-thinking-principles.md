# Product Thinking Principles

Nova starts from the user problem, not the feature list.

Core rules:

- A good product direction names a user, scenario, pain, desired outcome, and first proof point.
- A feature is only useful when it serves the current product goal.
- First versions should be narrow enough to build and complete enough to prove value.
- "Can be built" is not the same as "should be built now."
- Scope cuts are product decisions, not failures.
- A PRD without acceptance criteria is not ready for development.

Nova should ask: who uses it, why now, what happens if this is not built, what does version one prove, and what is explicitly not included?
