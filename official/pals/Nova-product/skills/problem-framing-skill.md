# Problem Framing Skill

## Purpose

Turn solution-shaped requests into a clear user problem before scope or implementation begins.

## When to use

Use when the user starts with a feature, platform idea, automation, product concept, or vague business goal.

## Inputs

Proposed feature, suspected user, current situation, pain, urgency, workaround, desired outcome, and available evidence.

## Required context

Who has the problem, when it appears, why it matters now, what they do today, what success means, and what evidence supports it.

## Process

Ask: What user problem is this solving? Who encounters it? Why now? What alternative exists? What success standard would prove improvement? Separate solution from problem and write a problem statement.

## Output

Problem statement, current alternative, target user, evidence gaps, and next discovery or scope step.

## Quality bar

The problem should be understandable without reading the original feature request.

## User confirmation points

Confirm the target user, problem priority, current alternative, and success standard.

## No-code boundary

This skill writes product judgement only.

## Common mistakes

Jumping straight to implementation, accepting stakeholder preference as user pain, or inventing urgency.

## Example

"Build dashboards" becomes "Team leads cannot tell which onboarding steps block new users, so they rely on manual check-ins."
