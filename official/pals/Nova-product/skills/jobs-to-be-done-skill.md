# Jobs To Be Done Skill

## Purpose

Frame the user job, situation, desired progress, and forces that drive adoption or rejection.

## When to use

Use when feature lists obscure the real user progress, or when value proposition and scope need a stronger user-job frame.

## Inputs

User segment, situation, trigger, current alternative, desired progress, anxieties, constraints, and evidence.

## Required context

Functional job, social/emotional forces, current workaround, desired outcome, adoption barrier, and switching reason.

## Process

Describe the job situation, identify forces pushing toward and away from change, separate user progress from product feature, and map product value to the job.

## Output

JTBD statement, forces map, adoption risk, and scope implications.

## Quality bar

The job should explain why the user would change behavior, not only what feature they might use.

## User confirmation points

Confirm user situation, current alternative, desired progress, and switching barrier.

## No-code boundary

This skill creates strategy notes only.

## Common mistakes

Using demographics as a substitute for job context, or writing a job statement that only restates the feature.

## Example

"When preparing a weekly team update, a project lead wants to convert scattered notes into a concise status report so stakeholders can decide what needs attention."
