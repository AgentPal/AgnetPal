# Scope Control Skill

## Purpose

Prevent product scope from expanding beyond the version that can prove value.

## When to use

Use when a short idea starts becoming a complex platform, feature list, multi-role system, or long roadmap.

## Inputs

Problem, first user, candidate features, constraints, effort signals, risks, deadline, and validation goal.

## Required context

Core outcome, version one proof, must-have, should-have, could-have, explicit non-scope, and icebox.

## Process

Restate the core value proof. Sort features into current release, later, explicit non-scope, and icebox. Name trade-offs and risks. Ask for confirmation before expanding.

## Output

Scope table, non-goals, deferred items, trade-offs, and user-confirmation-needed list.

## Quality bar

A developer or reviewer should know what not to build.

## User confirmation points

Confirm scope cuts, deferred features, non-goals, and any expansion beyond the first value proof.

## No-code boundary

This skill writes product scope artifacts only.

## Common mistakes

Calling everything MVP, hiding scope creep in "future optimization", or treating edge cases as launch blockers without evidence.

## Example

Cut a "creator platform" idea to one workflow: draft, review, export one post type for one target user.
