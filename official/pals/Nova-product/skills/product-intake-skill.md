# Product Intake Skill

## Purpose

Classify a product request and identify whether Nova should clarify, frame, scope, prioritize, write, review, or hand off.

## When to use

Use for every new Nova-owned request before product strategy, PRD, roadmap, MVP, or handoff work.

## Inputs

User idea, business goal, suspected user, requested feature, source evidence, constraints, desired artifact, and risk hints.

## Required context

Problem maturity, user clarity, evidence status, product stage, scope pressure, deadline, and expected next owner.

## Process

Classify the request as idea, problem, user scenario, requirement, PRD, priority, roadmap, release scope, risk review, metrics, or handoff. Identify missing problem, user, value, scope, evidence, risk, metric, and decision items. Select the smallest next Nova asset.

## Output

Product intake brief with maturity, missing evidence, selected next skill, likely deliverable, confirmation points, and collaboration needs.

## Quality bar

The brief should prevent premature implementation and make the next product step obvious.

## User confirmation points

Confirm target user, public claims, scope expansion, business decision, roadmap commitment, and handoff readiness.

## No-code boundary

This skill produces product briefs and Markdown task packages only.

## Common mistakes

Treating a feature request as a validated requirement, asking a long questionnaire, or hiding unknowns behind confident product language.

## Example

Input: "Add team analytics." Output: "Maturity: feature idea. Missing: who needs analytics, what decision it supports, current workaround, success metric."
