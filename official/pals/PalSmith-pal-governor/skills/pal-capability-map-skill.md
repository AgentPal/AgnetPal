# Pal Capability Map Skill

## Purpose

Map what Pals can actually do, including skills, knowledge, workflows, evidence, and gaps.

## When To Use

Use when the user asks what Pals can help with, when designing a team, or when checking whether a role is covered.

## Inputs

- Pal list or directory scope
- allowed registry/contacts reads
- target business/domain question
- whether private memory is excluded

## Required Context

Read target Pal root files, indexes, selected skills/knowledge/workflows, `pal-skill-vs-knowledge-vs-workflow.md`, and capability map protocol/template.

## Process

1. Identify candidate Pals from approved scope.
2. Read only public or approved assets.
3. Extract role, skills, knowledge, workflows, output contracts, evals, and collaboration modes.
4. Mark evidence source for each capability.
5. Separate proven capabilities from inferred capabilities.
6. List gaps and recommended PalSmith follow-up tasks.

## Output

- capability map table
- evidence source list
- gaps
- overlap notes
- recommended owner/verifier/consultant candidates

## Quality Bar

Every capability claim should point to a file or be labelled as inference. Missing evidence should be `missing` or `not-run`.

## User Confirmation Points

- reading private assets
- writing a report
- changing team design or contacts

## No-Code Boundary

PalSmith creates a map; it does not activate Pals or change contacts.

## Common Mistakes

- listing capabilities from names only
- reading every Pal by default
- mixing skills, knowledge, and workflows
- hiding gaps behind smooth summaries

## Example

For website operations, PalSmith can map Content Planner, SEO Reviewer, Publishing Checker, and Quality Verifier roles, then show where analytics knowledge is missing.
