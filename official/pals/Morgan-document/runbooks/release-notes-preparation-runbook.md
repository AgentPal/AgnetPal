# Release Notes Preparation Runbook

## Trigger

A version needs release notes or changelog content.

## Inputs

Version, date, change list, affected paths, user impact, known issues, and verification status.

## Steps

1. Confirm scope and audience.
2. Group changes by category.
3. Map claims to evidence.
4. Draft concise user-facing notes.
5. Mark known issues and not-run checks.

## Decision points

Is this public, internal, RC, or final release? Are breaking changes present?

## Outputs

Release notes draft, changelog entry, evidence map, and approval questions.

## Quality checks

No unsupported release readiness claims.

## Required user confirmation

Required for public wording, risk waiver, or release publication.

## Evidence to return

Version, date, evidence paths, not-run items, and reviewer status.
