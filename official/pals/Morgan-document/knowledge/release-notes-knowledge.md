# Release Notes Knowledge

## Concept Explanation

Release notes communicate notable changes, user impact, known issues, and verification state for a version.

## Judgement Standards

- Use version and date.
- Group changes in user-readable categories.
- Separate Added, Changed, Fixed, Removed, Deprecated, Security, Known issues, and Documentation when useful.
- Avoid raw commit dumps.

## Examples

- "Added Morgan Document / File Workflow Lead assets" with affected path summary and verification.

## Counterexamples

- "Many improvements" without specifics.
- Claiming full release readiness without release evidence.

## Scope

Changelogs, release summaries, RC reports, and docs update summaries.

## How This Becomes Skill, Workflow, Or Eval

Use in release-notes documentation skill, workflow, runbook, and eval.

## Common Mistakes

Overclaiming, omitting known limitations, and mixing internal notes with public notes.
