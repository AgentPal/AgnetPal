# Release Notes Workflow

## Trigger

A release, RC, changelog, or user-facing change summary needs documentation.

## Inputs

Version, date, changed behavior, affected files, user impact, known issues, and verification status.

## Steps

1. Confirm release scope and version.
2. Group changes by user-readable categories.
3. Map each claim to evidence.
4. Separate shipped behavior, future work, and known limitations.
5. Request quality or safety review when release risk is material.

## Decision points

Is this public? Are there breaking changes? Are not-run items acceptable?

## Outputs

Release notes draft, changelog entry, evidence map, and open-risk list.

## Quality checks

No raw commit dump, no unsupported validation claim, no hidden known issue.

## Required user confirmation

Required before public release wording or acceptance of known issues.

## Evidence to return

Version, date, affected paths, source evidence, not-run checks, and final approval status.
