# Document Quality Review Skill

## Purpose

Review whether a document or document task package is complete, readable, traceable, accessible, and evidence-backed.

## When to use

Use before accepting reports, README updates, Office/PDF outputs, release notes, or documentation handoffs.

## Inputs

Document draft or path, source inventory, acceptance criteria, output format, and evidence returned by Runtime.

## Required context

Target audience, required sections, source IDs, style expectations, accessibility expectations, and not-run items.

## Process

1. Compare document to original goal.
2. Check structure, headings, terminology, and source preservation.
3. Check completeness, contradictions, outdated notes, and unsupported claims.
4. Check output-format risks, accessibility, links, and evidence.
5. Return pass, partial, fail, not-run, blocked, or needs-more-evidence.

## Output

Quality review report with decision, evidence, findings, risk, not-run items, and next owner.

## Quality bar

No pass without source and output evidence.

## User confirmation points

Confirm acceptance of partial coverage, sensitive omissions, publication, or release-facing wording.

## No-code boundary

This skill reviews evidence and does not run automated validation.

## Common mistakes

Accepting completion claims without evidence, reviewing only grammar, or ignoring source loss.

## Example

Morgan reviews a release note draft and flags missing user impact, uncited breaking change, and absent version/date evidence.
