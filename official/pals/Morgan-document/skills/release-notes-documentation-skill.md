# Release Notes Documentation Skill

## Purpose

Prepare user-readable release notes or changelog entries from evidence.

## When to use

Use for release summaries, changelogs, upgrade notes, RC reports, known limitations, and user-facing change lists.

## Inputs

Version, date, changed files or features, bug fixes, breaking changes, migration notes, known issues, and evidence.

## Required context

Release scope, audience, source evidence, change categories, and unsupported claims list.

## Process

1. Confirm version and release date.
2. Group changes into added, changed, fixed, removed, deprecated, security, known issues, or documentation.
3. Write for users first, with evidence-backed claims.
4. Separate shipped behavior from planned work.
5. Ask Quinn or Rhea for quality/safety review when needed.

## Output

Release notes draft, changelog entry, evidence map, and review checklist.

## Quality bar

Every notable release claim must map to evidence or be removed.

## User confirmation points

Confirm public wording, breaking change severity, migration guidance, and unresolved risk.

## No-code boundary

This skill creates text only and does not publish releases.

## Common mistakes

Dumping raw commits, overclaiming validation, hiding known issues, or using internal-only notes publicly.

## Example

Morgan turns a file-change list into "Added", "Changed", "Known limitations", and "Verification" sections.
