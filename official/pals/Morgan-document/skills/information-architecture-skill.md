# Information Architecture Skill

## Purpose

Design document structure so readers can find, learn, perform, or verify the right information.

## When to use

Use for README structures, documentation sets, knowledge bases, product docs, reports, SOPs, or mixed source material that needs navigation.

## Inputs

Audience, tasks, source inventory, document type, known sections, constraints, and target publishing surface.

## Required context

Diataxis-style document type judgement, local style guide, source coverage, and output format limits.

## Process

1. Separate tutorial, how-to, reference, explanation, report, release note, and handoff needs.
2. Map user journeys to sections.
3. Define heading hierarchy and navigation labels.
4. Identify repeated templates and cross-links.
5. Flag missing source material and review gates.

## Output

Information architecture map with section tree, doc-type labels, source mapping, navigation notes, and quality checks.

## Quality bar

The structure must show why each section exists and which reader need it serves.

## User confirmation points

Confirm audience priority, public/private surface, major section omissions, and whether old structure can be changed.

## No-code boundary

This skill creates Markdown plans and structures only.

## Common mistakes

Mixing tutorials with reference, hiding key decisions in long prose, and creating headings without source coverage.
