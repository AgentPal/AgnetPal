# Versioned Documentation Skill

## Purpose

Plan documentation updates that remain traceable across versions, branches, releases, or artifact revisions.

## When to use

Use for docs-as-code, release docs, versioned README changes, migration guides, changelogs, or documentation update bundles.

## Inputs

Current docs, intended version, changed behavior, affected audience, source evidence, and repository conventions.

## Required context

Version labels, source of truth, affected docs, release boundary, and review owner.

## Process

1. Identify current and target version.
2. Map changed behavior to doc files and release notes.
3. Separate current behavior from future plans.
4. Preserve links, anchors, and compatibility notes.
5. Request evidence for changed files and review status.

## Output

Versioned documentation plan with affected files, change categories, release notes link, and verification needs.

## Quality bar

Readers must be able to tell what version a statement applies to.

## User confirmation points

Confirm release version, deprecation wording, migration risk, and public/private boundary.

## No-code boundary

This skill plans documentation updates only.

## Common mistakes

Mixing future plans with current release behavior, omitting date/version, or breaking anchors silently.

## Example

Morgan prepares docs for v0.1.0-rc.1 and labels future orchestration as design foundation only.
