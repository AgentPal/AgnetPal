# Markdown Documentation Skill

## Purpose

Plan and review Markdown documentation with clear structure, stable headings, links, lists, tables, and code fences when needed.

## When to use

Use for README files, docs folders, Markdown reports, runbooks, changelogs, specs, and AgentPal Pal assets.

## Inputs

Document goal, audience, source material, target renderer, repository conventions, and required sections.

## Required context

CommonMark or GitHub-flavored Markdown expectations, local docs style, and link targets.

## Process

1. Choose heading hierarchy.
2. Define section order and required tables.
3. Preserve link targets and file paths.
4. Use fenced code blocks only for literal snippets or commands.
5. Review for broken structure, duplicated headings, and inaccessible tables.

## Output

Markdown document plan, section template, review checklist, and Runtime evidence needs.

## Quality bar

The Markdown should be readable in source form and predictable in rendered form.

## User confirmation points

Confirm public/private content, generated source attribution, and any major restructuring of existing docs.

## No-code boundary

This skill writes or plans Markdown only, not rendering tools.

## Common mistakes

Using heading levels for visual size, mixing tables with long prose, or losing file-path accuracy.

## Example

Morgan turns a messy README request into title, summary, quick start, concepts, usage, troubleshooting, and verification sections.
