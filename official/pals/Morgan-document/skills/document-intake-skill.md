# Document Intake Skill

## Purpose

Classify a document or file-workflow request before any Runtime reads, rewrites, exports, or organizes files.

## When to use

Use when the user asks for a document, README, report, Office/PDF output, release note, source cleanup, folder plan, or document review and the scope is not yet fully bounded.

## Inputs

User goal, audience, source paths or pasted material, desired output format, deadline, privacy boundary, and evidence requirement.

## Required context

Morgan output contract, source inventory if present, destination format, allowed reads, forbidden reads, and whether user approval is required.

## Process

1. Identify document type and final deliverable.
2. Ask whether the task is authoring, restructuring, summarizing, preserving, reviewing, exporting, or handing off.
3. Mark sensitive material, copyright limits, and private paths.
4. Decide whether Morgan can own the stage or should prepare consult / delegate / handoff.
5. Produce an intake summary and acceptance criteria.

## Output

Document intake packet with scope, assumptions, source list, privacy boundary, output format, owner judgement, confirmation points, and evidence needs.

## Quality bar

The intake must be specific enough that a Runtime can act without guessing the source set or output target.

## User confirmation points

Confirm broad file reads, private document contents, external sharing, irreversible edits, publication, or sensitive-context handoff.

## No-code boundary

This skill writes no-code task packages only. It does not create tools, scanners, validators, converters, or automation.

## Common mistakes

Skipping source inventory, treating format as purpose, losing privacy boundaries, or assuming Morgan can execute file operations.

## Example

For "turn these notes into a PDF report", Morgan returns purpose, audience, source list, outline, PDF task package needs, approval points, and evidence checklist.
