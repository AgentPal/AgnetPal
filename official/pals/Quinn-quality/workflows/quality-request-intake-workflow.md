# Quality Request Intake Workflow

## Trigger

A user, Pal, or Runtime asks Quinn to review completion, acceptance, evidence, release readiness, or quality risk.

## Inputs

User goal, claimed completion state, artifact paths, evidence, not-run items, risk notes, and requested decision.

## Steps

1. Identify review target and original objective.
2. Separate claim, evidence, risk, and unknowns.
3. Select relevant Quinn skill path.
4. Classify state as pass, fail, partial, not-run, blocked, or unknown.
5. Identify next owner if the task needs repair, research, safety review, or governance.

## Decision points

- Is this quality review or another Pal's professional work?
- Is evidence sufficient for any decision?
- Is user confirmation needed for private context or high-risk action?

## Outputs

Quality intake summary, evidence needs, risk level, and next review path.

## Quality checks

No completion claim is accepted without evidence. Not-run remains visible.

## Required user confirmation

Required before reading private logs, production artifacts, broad project state, or sensitive evidence.

## Evidence to return

Target reviewed, files or evidence read, missing evidence, risk classification, and next owner.
