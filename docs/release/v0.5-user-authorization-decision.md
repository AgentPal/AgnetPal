## Actions That Need Separate Authorization

Each action is separate:

| Action | What it does | Requires explicit user authorization |
| --- | --- | --- |
| GitHub sync / push | Pushes local `master` commits to `origin/master` | yes |
| Local tag creation | Creates a local version tag on a chosen commit | yes |
| Tag push | Publishes a local tag to GitHub | yes |
| GitHub Release creation | Creates a GitHub Release page from a tag | yes |
| Release notes publication | Uses reviewed release notes as the GitHub Release body | yes |
| Release asset upload | Uploads a zip or other artifact to the release | yes |

Do not treat these as one button.

## Option A: Push Commits Only

Push current local commits to `origin/master`. Do not create a tag and do not create a GitHub Release.

Suitable when:

- the user wants the remote main branch synced first;
- the user wants a smaller, lower-risk step before version publication;
- the tag name and release notes still need final review.

Risk: low.

Tradeoff: no formal version marker and no GitHub Release page.

Example authorization:

```text
我授权执行 Option A：只 push 当前本地 commits 到 origin/master，不创建 tag，不创建 GitHub Release。
```

## Option B: Push Commits And Publish A Tag

Push current local commits, create a local tag on the approved commit, then push that tag. Do not create a GitHub Release.

Suitable when:

- the user wants a version marker before writing a GitHub Release;
- the final release page can wait;
- the tag name is confirmed.

Risk: medium.

Tradeoff: a pushed tag is public version history. Fixing it later needs careful human review.

Tag name needs user confirmation. Because `v0.5.0-rc.1` already exists locally on an older commit, safer candidates are:

- `v0.5.0-rc.2` for another release candidate;
- `v0.5.0` only if the user decides this is a stable release.

Example authorization:

```text
我授权执行 Option B：push 当前本地 commits，并创建 / 推送 tag <tag-name>，不创建 GitHub Release。
```

## Option C: Push Commits, Publish A Tag, And Create A GitHub Release

Push current local commits, create and push a confirmed tag, create a GitHub Release, publish release notes, and optionally upload an approved release asset.

Suitable when:

- the user wants an actual public GitHub Release;
- the version name is confirmed;
- release notes are approved;
- release asset policy is confirmed.

Risk: highest.

Tradeoff: this is the public release path. Any mistake may need release edits, tag remediation, or a follow-up release.

Before Option C, confirm:

- release tag name;
- release title;
- whether the release is stable or pre-release;
- whether any zip asset should be uploaded;
- which release notes body to use.

Example authorization:

```text
我授权执行 Option C：push 当前本地 commits，创建 / 推送 tag <tag-name>，并创建 GitHub Release。
```

## Recovery And Remediation

If only commits are pushed:

- Prefer follow-up commits or `git revert` for corrections.
- Do not rewrite remote history without explicit user authorization.

If a tag is pushed:

- Prefer a new corrected tag when possible.
- Deleting or replacing a remote tag is possible but high-risk and needs explicit user authorization.

If a GitHub Release is created:

- Edit the release body for wording mistakes.
- Mark as pre-release or draft if appropriate.
- Delete only with explicit user authorization and a clear replacement plan.

R195 does not perform any recovery action.
