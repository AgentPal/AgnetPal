# AgentPal v0.3 Development Plan

AgentPal v0.3 starts the no-code orchestration prototype line after the v0.2 first-phase release candidate.

The v0.3 goal is to move AgentPal from Pal assets and AI team usage paths toward controlled multi-Pal collaboration that host Agent Runtimes can follow through Markdown, JSON, protocol files, examples, evals, and task packages.

v0.3 does not make AgentPal an Agent Runtime. It does not add a UI, CLI, installer, scanner, validator, daemon, automatic external Agent calls, or automatic multi-runtime parallel execution.

## v0.3 Goal

v0.3 should make AgentPal a no-code orchestration prototype for complex work:

- direct `/pal Name` and `@Pal` collaboration behavior is explicit as a plain-text protocol, not a required native Runtime command;
- Context Packet becomes the standard handoff structure;
- Context Access Lists describe what a Pal, verifier, runtime, Skill, plugin, or MCP candidate may read and receive;
- Pal-owned Skills and Runtime-installed Skills are separated;
- Runtime Skill-aware Task Packages describe host Skill candidates without claiming AgentPal executes them;
- cross-runtime Pal memory keeps Pal continuity separate from current Runtime state;
- Cross-Runtime Pal Memory provides memory snapshots, routing records, Runtime Skill Usage Memory, continuation task packages, and privacy boundaries so the same Pal can continue a project across Codex, Claude Code, and generic CLI agents;
- Token / Cost-aware Conductor policy controls read depth, profile use, memory reuse, model/reasoning suggestions, and verification requirements;
- Deep Conductor Master Loop unifies project goal intake, project state judgement, task maps, topology selection, Runtime packages, verification, synthesis, and memory writeback;
- Project Conductor Workflow turns continuing project goals into no-code task maps and next-round packages;
- Owner + Verifier, Plan -> Execute -> Verify, and Parallel Independent Review become copyable staged protocols;
- Routing Memory writeback becomes a public-safe manual prototype;
- Memory writeback remains a host Runtime no-code task package, not an AgentPal database or sync service;
- PalBench Collaboration becomes the first regression suite for multi-Pal behavior.

## What v0.3 Does Not Do

v0.3 does not add:

- desktop UI or web UI;
- CLI, installer, scanner, validator, daemon, service, or background process;
- automatic multi-runtime parallel execution;
- automatic external Agent calls;
- Subagent Mode as default current ability;
- Deep Conductor as a fully automatic runtime;
- black-box scheduling;
- hard-coded task/domain/keyword routing.

All v0.3 collaboration artifacts are no-code staged protocols. The host runtime still performs real file reads, writes, commands, tool calls, publishing, rendering, and other execution, and must provide evidence.

## Module 1: Multi-Pal Mention / Direct Call Protocol

Current basis:

- `/pal Name` and `@Name` are recognized by current gates and contacts / registry.
- Mira is the ordinary entry Pal; directly called Pals apply the same core gates.
- Existing contacts and registry list allowed modes such as consult, delegate, handoff, review, and joint work.

Gap:

- current docs explain the concept but do not give a v0.3 collaboration protocol with Context Packet requirements, owner transfer rules, and consult defaults.

v0.3 target:

- define `/pal Name` as direct Pal work mode and `@Pal` as consult by default;
- define consult, delegate, handoff, review, and owner transfer;
- require Context Packet for collaboration;
- preserve direct owner Pal judgement when a specialist is called directly;
- allow a directly called Pal to name other Pal candidates or return conductor responsibility to Mira when the task exceeds its scope.

Acceptable artifacts:

- `orchestration/mention-and-direct-pal-protocol.md`;
- `templates/orchestration/direct-pal-call-template.md`;
- `templates/orchestration/mention-consult-template.md`;
- eval and PalBench Collaboration cases.

Boundary:

- no native slash-command runtime is created;
- no Pal is selected by keyword or fixed route;
- `@Pal` does not automatically start parallel work.

## Module 2: Context Packet / Context Access List Runtime Handoff

Current basis:

- Context Slicing, Context Access List, Task Package, and Mira output contracts already exist.
- Existing design states that index-known paths are not content-read evidence.

Gap:

- Context Packet is referenced but lacks a dedicated v0.3 protocol and copyable packet template.

v0.3 target:

- define Context Packet required fields;
- make `to_pal_candidate` a candidate field, not a fixed route;
- forbid full chat history transfer by default;
- separate `can_read`, `cannot_read`, and `can_receive_outputs_from`;
- keep private memory and sensitive context out by default.

Acceptable artifacts:

- `orchestration/context-packet-protocol.md`;
- `templates/orchestration/context-packet-template.md`;
- Context Packet examples and privacy-boundary evals.

Boundary:

- no automatic message bus;
- no sandbox or runtime permission grant;
- no full-workspace or full-chat forwarding.

## Module 3: Owner + Verifier Workflow

Current basis:

- Quinn owns quality and verification; Task Packages already require evidence.
- Workflow Topology lists Owner + Verifier as future design.

Gap:

- no concrete staged protocol, template, example, or self-test exists for v0.3 release-quality use.

v0.3 target:

- define owner Pal responsibilities for task judgement and task package;
- define verifier Pal responsibilities for independent evidence review;
- require verifier to read verification context rather than only the owner's conclusion;
- support pass, fail, partial, blocked, and not-run outcomes.

Acceptable artifacts:

- `orchestration/owner-verifier-workflow-protocol.md`;
- `templates/orchestration/owner-verifier-task-package.md`;
- release-check example;
- self-test.

Boundary:

- no automatic parallel execution;
- no verifier as final business approver;
- no pass without evidence.

## Module 4: Plan -> Execute -> Verify Workflow

Current basis:

- Pal Task Packages already separate Pal judgement from Runtime execution.
- Completion reports are already not evidence by themselves.

Gap:

- no dedicated staged workflow clarifies plan, execution evidence, and verification roles.

v0.3 target:

- define plan stage, execute stage, and verify stage;
- make Runtime an execution candidate, not a Pal-layer owner;
- require explicit evidence and not-run reporting;
- support release, docs, code, and PalSmith tasks.

Acceptable artifacts:

- `orchestration/plan-execute-verify-protocol.md`;
- `templates/orchestration/plan-execute-verify-package.md`;
- docs-update example;
- self-test.

Boundary:

- no runtime replaces Pal judgement;
- no automated execution loop;
- no completion claim from report text alone.

## Module 5: Parallel Independent Review Workflow

Current basis:

- Pal Isolation and Shared Memory protocol already names group-chat collapse and independent review principles.

Gap:

- no copyable review packet, example, or eval exists for isolated reviewer workflows.

v0.3 target:

- each reviewer receives a separate Context Packet;
- reviewers do not read each other's drafts;
- Mira or an owner Pal reads final reports only at the summary stage;
- conflicts, evidence quality, and recommended next actions are summarized explicitly.

Acceptable artifacts:

- `orchestration/parallel-independent-review-protocol.md`;
- `templates/orchestration/parallel-review-packet.md`;
- product / dev / QA example;
- self-test.

Boundary:

- no group chat;
- no automatic multithreading;
- no simple task inflation.

## Module 6: Routing Memory Writeback Prototype

Current basis:

- Routing Reward Memory protocol and routing decision templates already exist.
- v0.2 keeps routing memory manual and public-safe.

Gap:

- current protocol is reward-memory oriented but lacks v0.3 writeback flow, public-safe example records, and result record template.

v0.3 target:

- record routing decisions and results without private content;
- include task type, owner candidate, runtime candidate, reasoning level, context counts, verification result, and rework count;
- feed future AI Judgement as evidence, not fixed rules.

Acceptable artifacts:

- `orchestration/routing-memory-writeback-protocol.md`;
- `templates/orchestration/routing-decision-record.md`;
- `templates/orchestration/routing-result-record.md`;
- `memory/routing/README.md`;
- synthetic JSONL example;
- reuse example and self-test.

Boundary:

- no automatic database;
- no private memory in public examples;
- no "next time must route to X" rules.

## v0.3 And v0.4 Boundary

v0.3:

- no-code orchestration prototype;
- staged collaboration protocols;
- copyable Context Packets and workflow packages;
- manual public-safe routing memory records;
- PalBench Collaboration qualitative suite.

v0.4 default direction:

- Runtime Adapter Protocol Expansion;
- host Agent Runtime usage guides;
- Runtime-installed Skill orchestration examples;
- cross-runtime memory continuity;
- Deep Conductor patterns.

v0.4 is not the default route for runtime adapter automation, CLI, scanner, validator, installer, daemon, desktop app, web app, service, database, native router, or message bus work.

v0.5 default direction:

- Pal Pack ecosystem standards;
- quality checklists;
- Awesome Pal Packs style indexes;
- manual or Agent-executable certification protocol;
- community contribution templates.

v0.5 is not the default route for a programmatic Hub, marketplace backend, scanner, validator, installer, daemon, desktop app, web app, service, or database.

Future program features require an explicit product-direction change. They are not implied by v0.3 planning.

## Acceptance

v0.3 planning is acceptable when the roadmap, task pool, No-Code Future Boundary, Deep Conductor master goal, Pal Skill vs Runtime Skill protocol, Runtime Skill-aware Task Package, cross-runtime Pal memory protocol, Token / Cost-aware policy, Context Packet protocol, Mention / Direct Pal protocol, Owner + Verifier workflow, Plan -> Execute -> Verify workflow, Parallel Independent Review workflow, Routing Memory writeback prototype, and PalBench Collaboration suite all exist and preserve the no-code Pal layer boundary.
