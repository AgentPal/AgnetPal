# Release Notes Document Structure

## User Request

"Use the completion evidence to draft release notes and a maintainer handoff."

## When this Pal is a good candidate

Morgan is a good candidate because the task is structured release documentation from evidence.

## When this Pal should not take the whole task

Morgan should not decide release readiness alone. Quinn is a verifier candidate when pass/fail acceptance is requested.

## Task Judgement

- domain_focus: release documentation.
- deliverables: release notes structure and maintainer handoff.
- work_stages: collect evidence, organize changes, identify limitations, draft docs.
- capability_needs: documentation structure, content preservation, release-note clarity.
- pal_candidates: Morgan for document structure; Quinn candidate for readiness evidence review.
- runtime_candidates: file-editing Runtime if docs are written.
- verification_needs: source evidence, limitations, changed paths, not-run items.

## Context Needs

Need completion report, changed-file list, user-facing highlights, limitations, and release boundary.

## Suggested Task Package

- Goal: draft release notes from evidence.
- Output: highlights, changed files, known limitations, upgrade notes, maintainer handoff.
- Do not do: turn missing evidence into confident release claims.

## Good Response Example

Morgan structures release notes with evidence-backed highlights and clearly labeled limitations.

## Bad Response Example

Morgan converts a completion report into marketing copy and drops not-run checks.

## Verification / Acceptance Criteria

- Release notes are traceable to evidence.
- Limitations are preserved.
- Readiness claims are left to the verifier when needed.
