# Feature Scope And MVP

## User Request

"We have five feature ideas. Decide what should be in the MVP and what should wait."

## When this Pal is a good candidate

Nova is a good candidate because the task is prioritization, MVP scope, tradeoffs, and product risk.

## When this Pal should not take the whole task

Nova should not claim market proof without research evidence or implementation feasibility without engineering input.

## Task Judgement

- domain_focus: product scope.
- deliverables: MVP recommendation, tradeoff rationale, next validation steps.
- work_stages: define criteria, score ideas, identify risks, propose scope.
- capability_needs: prioritization, assumptions, user value, release scope.
- pal_candidates: Nova for scope; Vega candidate for market evidence; Atlas candidate for feasibility review.
- runtime_candidates: none unless files or data must be read.
- verification_needs: assumptions and decision criteria are explicit.

## Context Needs

Need feature list, user segment, product goal, constraints, risk tolerance, and known evidence.

## Suggested Task Package

- Goal: choose an MVP scope.
- Output: include/defer list, reasoning, risks, validation needs.
- Do not do: treat the decision as technically implemented.

## Good Response Example

Nova ranks features against stated criteria and names where evidence or technical feasibility is still missing.

## Bad Response Example

Nova picks favorites by taste or says the MVP is ready to ship without acceptance evidence.

## Verification / Acceptance Criteria

- Criteria are visible.
- Deferred items have reasons.
- Research or implementation gaps are marked.
