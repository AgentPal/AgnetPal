# Requirements Clarification

## User Request

"I want a dashboard for my users, but I am not sure what it should include. Help me clarify requirements."

## When this Pal is a good candidate

Nova is a good candidate because the task is product problem framing, user needs, scope, and requirements discovery.

## When this Pal should not take the whole task

Nova should not implement the dashboard or write final UI code. Implementation and verification need separate stage candidates.

## Task Judgement

- domain_focus: product requirements.
- deliverables: clarified goal, user segments, jobs, must/should/won't list.
- work_stages: clarify user, identify jobs, define scope, prepare handoff.
- capability_needs: problem framing, scope control, prioritization.
- pal_candidates: Nova for product framing; Atlas candidate for later implementation package; Quinn candidate for acceptance criteria.
- runtime_candidates: none for clarification.
- verification_needs: user confirms assumptions and priority.

## Context Needs

Need target users, problem, current workflow, constraints, business goal, and launch horizon.

## Suggested Task Package

- Goal: turn a vague dashboard idea into requirements.
- Output: user scenarios, must/should/won't, open questions, handoff notes.
- Do not do: start implementation without scope.

## Good Response Example

Nova asks focused questions, proposes a first scope, names assumptions, and separates product decisions from later implementation.

## Bad Response Example

Nova designs screens immediately, writes implementation steps, or treats the first feature idea as the final scope.

## Verification / Acceptance Criteria

- Requirements are user-centered.
- Scope boundaries are explicit.
- Implementation remains a later stage.
