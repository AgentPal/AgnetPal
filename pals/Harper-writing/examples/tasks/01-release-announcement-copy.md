# Release Announcement Copy

## User Request

"Write a GitHub release announcement from these confirmed changes and limitations."

## When this Pal is a good candidate

Harper is a good candidate because the task is audience-aware writing with tone, clarity, and fact boundaries.

## When this Pal should not take the whole task

Harper should not verify release readiness or invent unconfirmed improvements. Quinn or Morgan may be candidates for readiness or document structure.

## Task Judgement

- domain_focus: release copywriting.
- deliverables: announcement draft, headline options, limitation wording.
- work_stages: identify audience, preserve facts, shape message, self-review.
- capability_needs: writing craft, tone control, claim safety.
- pal_candidates: Harper for copy; Quinn candidate for release evidence review; Morgan candidate for release-note structure.
- runtime_candidates: none unless writing files is requested.
- verification_needs: every claim traces to provided facts.

## Context Needs

Need confirmed changes, known limitations, audience, tone, release channel, and words to avoid.

## Suggested Task Package

- Goal: draft release announcement from confirmed facts.
- Output: title, short summary, highlights, limitations, call to action.
- Do not do: add unsupported claims.

## Good Response Example

Harper writes polished copy that preserves the limitations and avoids pretending unverified work is complete.

## Bad Response Example

Harper turns the release into hype, drops known limitations, or adds unsupported performance claims.

## Verification / Acceptance Criteria

- Claims are source-backed by provided evidence.
- Tone matches audience.
- Limitations remain visible.
